# v0.2.0

This release tightens the convention into a validator-backed routing model with explicit repository and revision identity.

## Highlights

- renames the workstream target field to `domain_repo_url` so the spec models domain-owned routing instead of workstream-owned repositories
- adds explicit revision fields for routed enterprise and domain targets: `solution_git_ref` and `domain_git_ref`
- hardens the validator so topology-dependent rules are enforced in addition to schema validation
- aligns templates, examples, and starter artifacts with the updated canonical field set
- expands supporting reference material for brownfield adoption, machine access, and operational guidance

## Major Changes

### Routing model cleanup

The largest change in this release is the routing-model cleanup. This release standardizes on `domain_repo_url` as the domain-target field inside `domain-workstreams.yml`, and it removes the old workstream-scoped naming from the live convention. That change was made to eliminate ambiguity about whether a workstream owns a runtime boundary. The answer in this release is no: the domain owns the boundary.

### Revision identity

`initiatives.yml` and `domain-registry.yml` now carry explicit revision fields so a validator can confirm entrypoints against a concrete repository/revision pair:

- `solution_git_ref` for solution routing
- `domain_git_ref` for domain routing

That keeps the release consistent with the document's deterministic-routing claim.

### Validator hardening

`../skills/ea-convention/scripts/validate_convention.py` now enforces:

- `initiative_id` presence in `domain-workstreams.yml` when `initiatives.yml` exists
- `domain_repo_url` presence when no authoritative `domain-registry.yml` is available
- `domain_repo_url` equality between workstream and registry records when both are present
- migration warnings for legacy `workstream_repo_url` usage

## Notes For Adopters

- If you are migrating from older catalogs, keep the validator warnings enabled so legacy `workstream_repo_url` usage is visible during the transition.
- The new revision fields are part of the norm for routed enterprise and domain targets. Adopters should populate them rather than relying on implicit default branches.
